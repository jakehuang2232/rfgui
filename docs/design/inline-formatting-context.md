## 動機

現況 inline 排版是「per-run shape 後再 re-pack」：每個 Text / TextAreaTextRun / fragmentable Element 各自建一份 parley layout、把 wrap 好的 visual line 當 fragment 暴露，再由外層 solver 依寬度重新打包。代價是要自己在元素樹這層重算「行內還剩多少寬」，於是 seam 一多、heuristic 一個接一個：

- `first/full_available_width` 兩值串接（[measure.rs:60-138](../../src/view/layout/measure.rs:60)）
- `Text::first_line_width_for_word_boundary` 詞界補救（[text/inline_plan.rs:71](../../src/view/base_component/text/inline_plan.rs:71)）
- fragmentable Element 的 first-line width cap（[element/layout_trait.rs:429](../../src/view/base_component/element/layout_trait.rs:429)）
- `break_parley_lines` 的兩段寬度斷行（[text_layout.rs:625](../../src/view/text_layout.rs:625)）

這些不是個別 bug，而是架構的衍生補丁。最近一次具體症狀：續接 inline 元素（前一行尾端只剩狹小殘寬）的第一整行被 cap 成殘寬、擠進行尾截成 "Pe"，本文視覺上從第二行開始（「Inline Element Test」場景，[examples/bin/01_window/scene_windows/inline_test.rs:171](../../examples/bin/01_window/scene_windows/inline_test.rs:171)）。

parley 本身就是為「一份 layout 內混排 styled text + 原子 inline box」設計的；正規作法是讓每個 IFC 根用一份 parley layout，一次斷行——「行內還剩多少」是單一 pass 的內生狀態，詞放不下自然下移、原子盒放不下整顆下移，無需人工算殘寬。

## 正規模型（瀏覽器 IFC）

1. 把 block 底下**所有** inline-level 內容攤平成一條序列（各層 span 的文字、inline-block、圖片），全部進同一個 line-breaker。
2. 用 UAX#14 斷行規則跑**一次**。`overflow-wrap: anywhere` 的正確語意是「詞放不進**自己那一整行**才斷詞」，不是放不進剩餘空間就斷。
3. 斷完行後，再針對每個 inline 元素祖先，為它在每條線上覆蓋的片段畫 border/padding/背景（首片段加左 padding、末片段加右 padding）。這就是 CSS「inline 元素可跨行、邊框會斷開」。

分類：
- **inline 元素（span）**：文字直接參與父層的同一次斷行，本身不是會自己 wrap 的子 layout——只是事後的繪製分組。
- **inline-block / 圖片 / replaced / widget**：才是原子盒，整顆放不下就下移。

## IFC 分類規則（落地前置）

實作前先把 arena child 分成四類，避免 builder 在 traversal 時臨場猜：

| 節點 / 子樹 | IFC 內角色 | 備註 |
|---|---|---|
| `Text` / plain text run | text range | 文字內容直接 append 到 IFC root 的 backing string，style 以 byte range 記錄 |
| `Layout::Inline` 且可攤平的 `Element` | span / decoration group | 本身不再 measure/wrap；padding/border/background 事後由 per-line fragment rect 繪製 |
| 非攤平 inline-level child（badge、chip、button、圖片、replaced/widget） | atomic inline box | 先以 block-like constraints measure，尺寸餵給 `push_inline_box`，再由 parley 位置 place |
| block child 混入 inline | 暫不支援 | 先明確報錯或降級成 atomic block boundary；CSS anonymous block split 另案 |

「可攤平」的第一版判準應保守：只接受無 scroll / transform / absolute formatting context / 自身 block layout side effect 的 inline span。component boundary 可穿透，但穿透後仍依 resolved host node 分類；event routing 透過 byte range / inline_box id 回到來源節點，不讓 macro 或 component type 參與分類。

## 目標架構

IFC 根持有一份（或依硬換行 / block 邊界切成數份）parley layout：

1. **建構期**：走訪 inline 子樹，產出
   - parley 輸入：所有後代 inline 文字 → styled ranges（font/size/weight/color 逐 span push）；原子 inline-level 後代 → `push_inline_box`（需先 block-measure 拿到尺寸）。
   - **映射表**：`byte range ↔ 來源節點`、`inline_box index ↔ 來源節點`。這是 hit-test / caret / selection / 事件路由的根據。
2. **斷行**：parley 一次斷完。
3. **輸出**：
   - 走訪 line/run → glyph（TextPass 依 source byte range 回查 color/style map 分組；不可假設 parley run boundary 等於 style boundary）。
   - 原子 inline box → 用 parley 給的位置去 place 那些子節點（它們內部仍照 block layout）。
   - 依「inline 元素祖先 × 每條線」重建片段矩形 → 子 Element 從這些 rect 畫 bg/border/padding，不再自己 layout。

1. **封裝翻轉**：IFC 根需「穿透」inline 後代把文字 / 盒子拉進同一份 layout → inline 後代變「layout 被動」節點。需明確立 decision（比照 [text-not-is-a-element.md](text-not-is-a-element.md)）。
2. **parley inline_box 的 vertical-align**：parley 0.9.0 的 positioned inline box 接近 baseline-bottom 模型，可支援 default Element baseline = height；`Top/Middle/Bottom` 與未來 baseline-from-children 仍需 rfgui 自行加 offset，不可假設 parley 直接涵蓋。
3. **color/style span → glyph 分組與 raster handle**：TextPass 需依 source byte range 回查 style map 切片段；parley run 可能因 shaping / bidi / itemization 分裂或合併，不可當作 style boundary。IFC glyph payload 也必須保留 `FontData` 與 normalized coords hash，否則只能描述 font identity，無法直接走既有 glyph raster cache key。
4. **block-in-inline**：block 子節點混進 inline → CSS 的 anonymous block 切分。初期**限制不支援**、明確報錯，之後再做。
5. **TextArea 邊界**：P0–P6 只建立 IFC engine 與 read-only primitives；editable IFC、IME、projection、caret affinity、scroll follow 等放 P7，避免把 TextArea 大遷移塞進 inline engine MVP。
6. **效能**：單根單 shape；大文件（TextArea）沿用 chunking + cache。
7. **回歸面**：examples / components 依賴現行 inline 行為的全跑一次。
8. **OverflowWrap**：現況 `break_parley_lines` 無條件開 `OverflowWrap::Anywhere`（[text_layout.rs:575/610](../../src/view/text_layout.rs:575)），是逼出 mid-word 斷字、進而需要詞界 heuristic 的元兇之一。統一後應收斂成 CSS 語意（只在詞放不進整行時才斷）。

## 不變的部分

- parley adapter（`text_layout.rs`）——擴充，不替換。
- flex / flow solver 的對外語意——不變；shared helper 可為 IFC path 拆分。
- vertical-align D1–D4 度量——重用。
- TextArea caret_map 概念——重用；正式合併留 P7 評估。

## 短期與長期的關係

短期的 width-cap 修正（+ 收斂 `OverflowWrap::Anywhere`）是「人工模擬單一 pass 結果」的正確局部補丁，是過渡橋；P0–P6 落地後由 P6 一併移除。長期主線是 P0–P6；P7（TextArea）獨立評估。
